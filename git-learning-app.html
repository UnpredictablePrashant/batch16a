Git Learning — Start Here

A minimal guide from GitHub docs. No jargon, just what you need to start.

What is Git?

Git is a version control system. It keeps a history of your project so you can:

  • See what changed, when, and who changed it
  • Go back to any earlier version if needed
  • Work with others on the same project without overwriting each other

A repository (repo) is your project folder plus its full history. GitHub hosts repos on the web so you can share and collaborate.

Essential commands (plain English)

CommandWhat it does
git initTurn the current folder into a Git repo
git clone <url>Download a repo from GitHub to your computer
git statusSee which files changed (new, modified, staged)
git add <file>Stage changes (prepare them for the next snapshot)
git commit -m "message"Save a snapshot with a short description
git pushUpload your commits to GitHub
git pullDownload latest changes from GitHub and merge into your branch
git fetchDownload latest from GitHub but don’t merge yet (lets you inspect first)
git logShow commit history (who, when, message). Use git log --oneline for a short list
git merge <branch>Combine another branch into your current branch (e.g. merge feature-a into main)
git rebase <branch>Replay your commits on top of another branch (linear history; use with care on shared branches)
git stashTemporarily save uncommitted changes so you can switch branches; git stash pop brings them back
git branchList branches; git checkout <branch> switches to one

Fetch vs Pull — what’s the difference?

Both download new commits from GitHub. The difference is what happens to your current branch.

  • git pull = download and merge into your branch in one step. Your files update immediately.
  • git fetch = download only. Your working files don’t change yet; you can look at what arrived, then decide to merge (or not).

When to use pull

You just want the latest from main and you’re fine with merging right away:

git checkout main
git pull

When to use fetch

You have local work and want to see what’s on GitHub before changing your branch. For example: “Did someone push to main? Let me check first.”

# Download latest from GitHub (your files don’t change yet)
git fetch origin

# See what’s new on main (compare your branch to origin/main)
git log main..origin/main --oneline

# If it looks good, now merge those changes into your current branch
git merge origin/main

Rule of thumb: use pull when you’re okay with “just get the latest.” Use fetch when you want to look before merging (e.g. you have uncommitted changes or you’re on a feature branch).

Other commands — when to use them

git stash

Scenario: You’re in the middle of edits on feature-x. Someone asks you to quickly fix a bug on main. You don’t want to commit half-done work.

git stash
git checkout main
# … fix the bug, commit, push …
git checkout feature-x
git stash pop

Your previous edits reappear and you can continue. Use git stash list if you have multiple stashes.

git log

Scenario: “What changed recently? Who added this line?”

git log --oneline -10
# Or see which commits touched a specific file:
git log --oneline -- README.md

merge vs rebase

Merge = “Add a merge commit that joins two branches.” History shows both branches; safe and clear for shared branches (e.g. merging a feature into main).

Rebase = “Replay my commits on top of another branch.” History looks like a straight line; nice for cleaning up before a PR. Avoid rebasing branches that others are already using (it rewrites history).

When in doubt, use merge. Use rebase only on your own branch (e.g. git rebase main to update your feature branch with latest main before opening a PR).

Practical example 1 — Start a new repo and put it on GitHub

First create an empty repo on GitHub (no README). Then run these in a terminal (same folder as your project):

git init my-repo
cd my-repo

# Create a file (e.g. README)
touch README.md

# Tell Git to track it and save first snapshot
git add README.md
git commit -m "add README to initial commit"

# Link to your GitHub repo (replace with your URL)
git remote add origin https://github.com/YOUR-USERNAME/YOUR-REPO-NAME.git

# Upload to GitHub
git push --set-upstream origin main

Practical example 2 — Get someone else’s repo and contribute

Download the repo, make a branch, edit files, then push your branch so you can open a Pull Request on GitHub.

# Download the repo (replace with real URL)
git clone https://github.com/owner/repo.git
cd repo

# Create and switch to a new branch
git branch my-branch
git checkout my-branch

# Edit files (e.g. file1.md, file2.md), then:
git add file1.md file2.md
git commit -m "my snapshot"
git push --set-upstream origin my-branch

Then on GitHub: open a Pull Request from my-branch to ask maintainers to merge your changes.

Practical example 3 — Update your copy and work on an existing branch

You already have the repo locally; new work was pushed to a branch like feature-a. Stay in sync and push your edits.

cd repo

# Get latest from GitHub
git pull

# Switch to the branch you work on
git checkout feature-a

# After editing file1.md:
git add file1.md
git commit -m "edit file1"
git push

GitHub flow in one sentence

Create a branch → make changes → commit and push → open a Pull Request → after review, merge and (optionally) delete the branch.
